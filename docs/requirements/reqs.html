Функциональные требования

User Story I:​

Как ученик, я хочу не просто просматривать задания, а иметь возможность фильтровать их по разным критериям.

FR-11:​

На страницу поиска должен быть интегрирован модуль фильтра по следующим параметрам:

  1. Номер задания (TaskNum) - ComboBox с элементами вида "Задание {num}", где num - число 1-27.

  2. Тема внутри задания (Theme) - ComboBox с элементами вида "<CheckBox> {num}.{idx} - {name}", где num - номер задания, idx - номер подтемы, name - текстовое название темы. Поля idx и name следует брать из соответствующих колонок из БД. <CheckBox> позволяет выбрать пользователю одну или более тем.

  3. Сложность (Diff) - ComboBox с элементами вида "<CheckBox> {dif}", где dif - сложность (от простого к сложному), берется из БД. Наличие <CheckBox> у каждого элемента позволяет выбрать пользователю один или несколько сложностей.

Если какой-либо из параметров не был выбран, он не должен влиять на результат поиска. Например, если пропущен критерий "Сложность", выводить следует все элементы соответствующего поля.

User Story II:

Как ученик, я хочу иметь возможность отслеживать свой прогресс по заданиям в личном кабинете. Мне нужна статистика по проценту успешно решенных заданий и по темам.

FR-21:​

Платформа должна обладать модулем регистрации, авторизации и восстановления. Модуль действует на странице аутентификации, состоящей из вкладки "войти" и "зарегистрироваться". Регистрация должна производиться по почтовому адресу и паролю. Пароль может содержать буквы (A-Z), цифры (0-9) и набор специальных символов {!, -, _, .}. Пароль принимается при выполнении следующих требований:

  1. Пароль должен состоять минимум из 10 символов.

  2. В пароле должна содержаться минимум по одной букве в нижнем и верхнем регистре.

  3. В пароле должна содержаться минимум одна цифра.

  4. В пароле должен содержаться минимум один специальный символ из набора {!, -, _, .}.

После выбора пароля пользователь должен ввести имя и фамилию, которые будут отображаться в личном кабинете. Эти поля могут состоять только из кириллицы (А-Я).

Авторизация производится по совокупности почты и пароля.

Восстановление осуществляется по нажатию соответствующей кнопке на этапе авторизации. Для успешного восстановления пользователь должен ввести почту, которую он указывал при регистрации. После этого ему на почту приходит письмо с индивидуальной ссылкой, перейдя по которой, он может придумать новый пароль.

Основной поток 1
(регистрация)		1. Пользователь нажимает на кнопку "войти" и попадает на страницу авторизации.
2. На странице аутентификации пользователь переходит на вкладку регистрации.
3. Пользователь вводит почту и пароль, нажимает кнопку "Зарегистрироваться!".
4. Сервис отправляет запрос в БД, чтобы проверить, зарегистрирован ли пользователь с такой почтой.
5. Если нет, то сервис добавляет в БД пользователя (почта и хэш пароля).
6. Сервис загружает страницу личного кабинета пользователя.
Альтернативный поток 1Точка входа: шаг 4.
4.1. Если в БД уже есть запись с таким логином (почтой), система выводит на странице сообщение о том, что такой пользователь уже существует.
4.2. Пользователь переходит на вкладку "войти" и выполняет авторизацию.
Основной поток 2
(авторизация)		1. Неавторизованный пользователь нажимает кнопку "войти" и попадает на страницу авторизации.
2. Пользователь вводит логин (почту) и пароль, указанные при регистрации.
3. Сервис проводит идентификацию:
3.1. Сервис отправляет запрос в БД и проверяет, есть ли в системе пользователь с таким логином.
3.2. Сервис получает положительный ответ и сравнивает введенный пароль с тем, который хранится в БД.
3.3. Если пароли (а точнее хэш) совпадает, сервис открывает страницу личного кабинета пользователя.
Альтернативный поток 2Точка входа: шаг 3.
- Если на этапе 3.1. запись в БД не была обнаружена, значит пользователя с таким логином нет в системе. Система выводит соответствующее сообщение на странице. Затем пользователь проходит регистрацию.
- Если на этапе 3.3. пароли не совпали, сервис показывает на странице соответствующее сообщение. Затем пользователь проходит процедуру восстановления.
Основной поток 3
(восстановление)		1. На вкладке "войти" пользователь нажимает кнопку "восстановить пароль".
2. Сервис открывает страницу восстановления, на которой пользователь должен ввести свой логин (почту).
3. Сервис отправляет в БД запрос и проверяет, существует ли пользователь с таким логином.
4. Если ответ положительный, то сервис формирует индивидуальную ссылку и отправляет ее по указанной почте. Ссылка действительно в течение 24 часов.
5. Пользователь открывает почту и переходит по ссылке.
6. Сервис загружает форму для ввода нового пароля.
7. Пользователь вводит пароль и подтверждает его (вводит повторно).
8. Сервис снова проверяет существование данного пользователя.
9. Если существует, то меняет пароль в соответствующей записи БД.
10. Сервис показывает сообщение об успешной смене пароля и предлагает пользователю пройти авторизацию в системе.
Альтернативный поток 3.1Точка входа: шаг 3.
3.1. Если такого пользователя нет в БД, сервис показывает на странице соответствующее сообщение.
3.2. Пользователь проходит регистрацию.
Альтернативный поток 3.2Точка входа: шаг 6.
6.1. Если с момента генерации ссылки прошло более 24 часов, сервис показывает сообщение о том, что ссылка недействительна.
6.2. Пользователь заново проходит процедуру восстановления.
Альтернативный поток 3.3Точка входа: шаг 8.
8.1. Если по какой-то причине пользователь не был найден в БД, то сервис выводит сообщение "что-то пошло не так - указанный пользователь не найден".
8.2. Либо пользователь повторяет процедуру восстановления, либо переходит на этап регистрации.
Исключительный поток 1Если на каком-либо из этапов не пришел ответ из БД, сервис показывает на странице сообщение "сервис временно не доступен, повторите попытку позже".

FR-22:​

Платформа должна обладать личным кабинетом (ЛК). На главной странице ЛК приведен график статистики выполненных заданий. Ось абсцисс - номер задания, ось ординат - процент выполнения, в процентных пунктах. График загружается на страницу в виде изображения. При нажатии ЛКМ на изображение оно открывается на всю страницу и пользователь должен иметь возможность масштабировать его (с помощью колесика мыши и клавиши CTRL).

Постусловия (неудача)Отображается сообщение об ошибке
Основной поток1. Пользователь нажимает на кнопку "ЛК".
2. Сервис загружает страницу личного кабинета:
2.1. Сервис обращается к БД для получения следующих данных:
- Номера заданий.
- Количество правильно решенных задач по каждому заданию.
- Количество всех решенных задач по каждому заданию.
2.2. На основе этих данных сервис вычисляет процент правильного выполнения задач по каждому заданию и строит график:
- Каждая точка на графике графически обозначается окружностью.
- Соседние точки графика (по оси абсцисс) соединяются прямой пунктирной линией.
2.3. График сохраняется в виде изображения.
2.4. Изображение загружается на страницу.
3. Пользователь видит страницу ЛК со статистикой.
Альтернативный потокТочка входа: 2.2.
Если пользователь не решил ни одно задание с номером k, то точка с соответствующей абсциссой k отсутствует на графике. Точки с абсциссами (k-1) и (k+1) считаются соседними и соединяются прямой пунктирной линией.

Например, если пользователь не решил ни одного задания №5, то на графике не будет точки с абсциссой, равной 5. При этом точки с абсциссами 4 и 6 (при их наличии) будут соединены прямой пунктирной линией.
Если при всем при этом отсутствуют данные по заданию номер 6, то соединяются точки с абсциссами 4 и 7. И так далее.
Исключительный поток 1Точка входа: шаг 2.
Если от БД не пришел ответ, то сервис показывает сообщение о том, что личный кабинет временно недоступен.

Логи хранятся в течение 90 дней, а затем подлежат удалению.		152-ФЗ о ПДн.

Последствия: репутационные издержки + административная и уголовная ответственность.

NFR-04Надежность/безопасностьВыполнять резервную копию пользовательских данных из БД каждый день в часы наименьшей нагрузки (02:00 МСК).152-ФЗ о ПДн.

⋅ Размер хранилища для данных БД - 5 ТБ с возможностью расширения до 10 ТБ.		Объем данных для одного пользователя не превышает 10 МБ. Для 100 тыс. пользователей достаточно 1 ТБ, для 1 млн. - 10 ТБ.
